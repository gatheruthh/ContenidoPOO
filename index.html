Contexto: Evolución del Software

El desarrollo de software ha pasado por tres grandes etapas de abstracción:

1. Lenguaje Máquina y Bajo Nivel

Programación basada en binario y Ensamblador. Control directo del hardware pero extremadamente difícil de mantener.

; Ejemplo Ensamblador x86
MOV AX, 5
ADD AX, 10
INT 21h

2. Lenguaje de Alto Nivel (Estructurado)

Basado en algoritmos y procedimientos secuenciales. Las funciones y los datos están separados.

// Ejemplo C# Estructurado
static void Calcular(float n1, float n2) {
  float r = n1 + n2;
  Console.WriteLine(r);
}

La POO surge como respuesta a la complejidad, permitiendo que el programador piense en "entidades" y no solo en líneas de ejecución.

¿Qué es la POO?

Es un modelo de programación en el que el diseño de software se organiza en torno a datos u objetos, en lugar de funciones y lógica.

Un objeto se define como un campo de datos que tiene atributos y comportamiento únicos.

Un Objeto posee:

  • Estado: Los datos que almacena (Atributos).
  • Comportamiento: Las acciones que puede realizar (Métodos).
  • Identidad: Lo que lo hace único en memoria.

Representación visual de la encapsulación: Datos + Lógica en una sola unidad.

La Clase: El Molde de Creación

La Clase (El Plano)

Es una plantilla abstracta. Define qué atributos y métodos tendrán los objetos creados a partir de ella, pero no ocupa espacio real de datos.

class Carro {
  public string color;
  public int velocidades;

  public void Acelerar() {
    Console.WriteLine("Avanzando...");
  }
}

El Objeto (La Instancia)

Es la entidad física en memoria. Se crea mediante la Instanciación usando un Constructor.

// Instanciación
Carro miCoche = new Carro();

// Asignación de estado
miCoche.color = "Rojo";
miCoche.velocidades = 6;

// Ejecución de comportamiento
miCoche.Acelerar();

Ejemplo 5: Sistema de Nómina UTHH

Caso real de aplicación en la Universidad Tecnológica de la Huasteca Hidalguense.

class ProfesorTC {
  string nombre = "Juan Perez";
  double sueldoBase = 6000.00;
  double apoyoDidactico = 800.00;
  double isr = 960.00;

  public void CalcularNomina() {
    double neto = (sueldoBase + apoyoDidactico) - isr;
    Console.WriteLine("Pago: " + neto);
  }
}

Análisis del Objeto Nómina

Estado: Contiene las variables financieras del docente.

Comportamiento: El método CalcularNomina() procesa los datos internos para generar un resultado.

Instancias Diferentes:
- Juan: Pago $5,840.00
- Karen: Pago $5,540.00 (Atributos distintos)

Pilares de la POO

1. Abstracción

2. Encapsulamiento

3. Herencia

4. Polimorfismo

1. Abstracción

Es la capacidad de enfocarse en las características esenciales de un objeto, ignorando los detalles complejos de implementación que no son relevantes para el usuario.

Analogía del Coche: Un conductor solo necesita conocer la interfaz (volante, pedales). No necesita saber cómo la inyección electrónica dosifica el combustible.

Aplicación en Software

En el sistema de nómina de la UTHH, un usuario solo ingresa datos. La complejidad matemática del cálculo de impuestos está abstraída dentro de la clase.

2. Encapsulamiento

Es el mecanismo que oculta los datos internos de un objeto, protegiéndolos de accesos externos no autorizados.

Se logra mediante modificadores de acceso:

  • Public: Accesible desde cualquier lugar.
  • Private: Solo accesible dentro de la clase.
  • Protected: Accesible por la clase y sus herederas.
class CuentaBancaria {
  private double saldo; // Oculto

  public void Depositar(double monto) {
    if(monto > 0) saldo += monto;
  }

  public double VerSaldo() {
    return saldo;
  }
}

3. Herencia

Permite definir una clase nueva a partir de una existente, heredando sus atributos y métodos.

// Clase Padre
class Trabajador {
  public string nombre;
  public double sueldoBase;
}

// Clase Hija (Hereda de Trabajador)
class Administrativo : Trabajador {
  public double bonoSeguro;
}

Jerarquía UTHH

Clase Base: Trabajador

Clases Derivadas:

  • ProfesorTC (Hereda nombre y añade apoyoDidactico).
  • Administrativo (Hereda nombre y añade bonoSeguro).

Evita la redundancia de código al no tener que redefinir "nombre" o "RFC" en cada tipo de empleado.

4. Polimorfismo

Literalmente significa "muchas formas". Es la capacidad de que objetos de diferentes clases herederas respondan al mismo método de formas distintas.

Ejemplo: El método CalcularSueldo() se comporta distinto si el objeto es un Profesor (suma horas) o un Administrativo (suma bonos fixed).

virtual void CalcularSueldo() { /* Base */ }

// En clase Profesor
override void CalcularSueldo() {
  sueldo = base + apoyoDidactico;
}

// En clase Administrativo
override void CalcularSueldo() {
  sueldo = base + bono;
}
